# Рекомендательная система

## Подготовка окружения

```bash
python3 -m venv env_recsys_start
. env_recsys_start/bin/activate
pip install -r requirements.txt
```

Скачайте файлы данных в директорию проекта:

```bash
wget https://storage.yandexcloud.net/mle-data/ym/tracks.parquet
wget https://storage.yandexcloud.net/mle-data/ym/catalog_names.parquet
wget https://storage.yandexcloud.net/mle-data/ym/interactions.parquet
```

## Расчет рекомендаций

Ноутбук `recommendations.ipynb` содержит код для подготовки данных и расчета офлайн-рекомендаций. Результаты сохраняются в S3.

## Архитектура сервиса рекомендаций

Система состоит из трех микросервисов:

| Сервис | Файл | Порт | Назначение |
|--------|------|------|------------|
| Features Store | `features_service.py` | 8000 | Хранение и выдача похожих товаров (item-to-item) |
| Events Store | `events_service.py` | 8001 | Хранение онлайн-событий пользователей |
| Recommendations | `recommendation_service.py` | 8002 | Выдача рекомендаций (офлайн, онлайн, комбинированные) |

## Запуск сервиса рекомендаций

Запустите все три сервиса в отдельных терминалах:

```bash
# Терминал 1: сервис фичей (порт 8000)
uvicorn features_service:app --port 8000

# Терминал 2: сервис событий (порт 8001)
uvicorn events_service:app --port 8001

# Терминал 3: сервис рекомендаций (порт 8002)
uvicorn recommendation_service:app --port 8002
```

## Стратегия смешивания онлайн- и офлайн-рекомендаций

Сервис предоставляет три эндпоинта:

- **`/recommendations_offline`** -- офлайн-рекомендации. Если для пользователя есть персональные рекомендации (предрасчитанные в ноутбуке), возвращаются они. Иначе возвращаются дефолтные рекомендации (топ популярных треков).

- **`/recommendations_online`** -- онлайн-рекомендации. На основе последних взаимодействий пользователя (из Events Store) для каждого просмотренного трека запрашиваются похожие треки (из Features Store). Результаты объединяются и сортируются по score.

- **`/recommendations`** -- комбинированные рекомендации. Используется стратегия чередования (interleaving):
  1. Получить офлайн-рекомендации и онлайн-рекомендации.
  2. Чередовать элементы: сначала онлайн[0], затем офлайн[0], затем онлайн[1], офлайн[1] и т.д.
  3. Добавить оставшиеся элементы из более длинного списка.
  4. Удалить дубликаты.
  5. Вернуть топ-k результатов.

Такая стратегия позволяет учитывать как долгосрочные предпочтения пользователя (офлайн), так и его текущие интересы (онлайн). Приоритет отдается онлайн-рекомендациям -- они идут на нечетных позициях (1, 3, 5...), что обеспечивает актуальность выдачи. Если у пользователя нет онлайн-истории, комбинированные рекомендации совпадают с офлайн.

## Тестирование сервиса

Убедитесь, что все три сервиса запущены (см. раздел "Запуск"). Затем выполните:

```bash
python test_service.py
```

Вывод автоматически сохраняется в файл `test_service.log`.

Тестовый скрипт проверяет следующие сценарии:

| Тест | Сценарий | Ожидаемый результат |
|------|----------|---------------------|
| 1 | Сервис фичей: похожие товары | Возвращает список похожих item_id с оценками |
| 2 | Сервис событий: сохранение и получение | События сохраняются и возвращаются корректно |
| 3 | Пользователь без персональных рекомендаций | Возвращаются дефолтные рекомендации (топ популярных) |
| 4 | Пользователь с персональными рекомендациями, без онлайн-истории | Комбинированные = офлайн (онлайн пустой) |
| 5 | Пользователь с персональными рекомендациями и онлайн-историей | Комбинированные чередуют онлайн и офлайн |
